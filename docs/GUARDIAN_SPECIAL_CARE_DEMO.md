# 家庭守护与特殊关怀模式演示指南

## 一、功能概述

### 家庭守护模式
家庭守护模式解决的是：当用户出现状态异常时，如何让监护人及时知晓并给予关怀的问题。

传统方案通常是简单的「高风险弹窗」，但这种方案存在以下问题：
- 频繁打扰监护人，导致疲劳
- 无法区分真实异常和短暂波动
- 缺乏闭环反馈机制
- 监护人无法了解具体情况

**我们的方案**：从状态偏离识别、监护人提醒、远程反馈、策略调参到闭环记录的一整套守护流程。

### 特殊关怀模式
特殊关怀模式针对需要更细致照护的用户，提供：
- 多维度状态偏离分级（轻微偏离 → 需要观察 → 建议照护确认 → 重点关注）
- 基于历史数据的连续异常检测
- 运动状态自动降级，避免误判
- 照护闭环状态跟踪

### 为什么不是医学诊断
本产品**不做医学诊断**，原因如下：
1. 监测数据来源于消费级可穿戴设备，精度有限
2. 心理/情绪状态受多因素影响，单一指标不足以判断
3. 医学诊断需要专业医师资质

**我们提供的是**：基于日常行为模式的状态偏离提示，作为家属关怀的辅助参考。

## 二、演示前准备

在 NeuroGarden App 中打开「家庭守护」页面。

### 步骤 2：填写监护人信息
- 监护人姓名：张叔叔
- 关系：父亲
- 联系方式：（可选填，用于后续真实通知）

### 步骤 3：选择通知渠道
默认选择「App 模拟通知」，点击确认授权。

### 步骤 4：开启特殊关怀模式（如需要）
在家庭守护设置中打开「特殊关怀模式」开关。

## 三、演示流程

### 步骤一：进入异常事件详情
从「历史记录」或「实时监测」进入任意一个异常事件详情页。

### 步骤二：查看状态偏离等级
在事件详情页中查看「状态偏离等级」，可能显示：
- 轻微偏离：单次轻度波动
- 需要观察：多次相似波动或中等偏离
- 建议照护确认：连续异常或多指标偏离
- 重点关注：多次严重偏离或监护人主动要求

### 步骤三：点击模拟通知监护人
点击「模拟通知监护人」按钮，系统会：
1. 根据当前策略判断是否应该通知
2. 生成模拟通知记录
3. 显示「当前为模拟通知，不会真实发送」提示

### 步骤四：查看通知记录
在家庭守护页底部查看「通知记录」，可以看到：
- 通知时间
- 通知原因
- 当前状态（已发送/已跳过）

### 步骤五：点击监护人反馈
点击任意一条通知记录，选择反馈选项：
- 已联系本人
- 继续观察
- 标记误报
- 提高/降低优先级

### 步骤六：查看照护闭环状态变化
反馈后，照护闭环状态会相应变化：
- 待观察 → 已提醒（通知发出后）
- 已提醒 → 已确认（收到反馈）
- 已提醒 → 继续观察（选择继续观察）
- 已提醒 → 误报（标记为误报）
- 继续观察 → 已处理（联系本人后）

### 步骤七：展示系统调参结果
多次反馈后，系统会自动调整策略：
- 标记误报 → 降低同类事件通知频率
- 提高优先级 → 降低触发阈值
- 降低优先级 → 提高触发阈值

## 五、隐私说明

本产品高度重视用户隐私：

1. **不上传输入原文**：用户输入的文本内容不会被上传
2. **不共享聊天内容**：不读取任何即时通讯软件数据
3. **不做医学诊断**：明确标注为非医学诊断，仅供参考
4. **只共享结构化摘要**：仅共享时间、偏离指标、风险等级等结构化数据
5. **用户可撤销授权**：随时可以在设置中撤销监护人授权

## 六、后续真实通知服务接入方式

当前系统使用 Mock 通知发送器，接入真实服务非常简单：

### 1. 替换发送器实现

```kotlin
// 当前默认使用 MockSender
val sender = GuardianNotificationSenderFactory.getDefaultSender()

// 后续替换为真实发送器
val smsSender = SmsGuardianNotificationSender()
val emailSender = EmailGuardianNotificationSender()
val wechatSender = WechatGuardianNotificationSender()
```

### 2. 配置短信服务（示例）

```kotlin
class SmsGuardianNotificationSender : GuardianNotificationSender {
    // 通过系统配置获取 API 密钥，不要硬编码
    private val apiKey = System.getenv("SMS_API_KEY")

    override suspend fun send(notification: GuardianNotificationPayload): GuardianNotificationSendResult {
        // 调用短信 API
    }
}
```

### 3. 保留原有策略逻辑

所有现有策略（shouldNotifyGuardian）保持不变：
- 连续异常检测
- 夜间优先
- 高心率低运动检测
- 误报频率限制
- 冷却期控制

### 4. 保留历史记录

通知历史（notificationHistory）和反馈历史（feedbackHistory）会继续记录所有发送结果，用于策略调参。

## 八、测试覆盖

当前已实现的单元测试覆盖：

1. ✅ 未授权时不允许真实通知
2. ✅ 本地模拟体验可以生成 mock notification
3. ✅ 低风险事件不通知监护人
4. ✅ 中高风险事件满足条件时通知
5. ✅ 连续多次异常才升级通知
6. ✅ 夜间异常优先通知
7. ✅ 运动状态下降级
8. ✅ 高心率 + 低运动优先提醒
9. ✅ 已联系本人后冷却期内不重复提醒
10. ✅ 标记误报后同类通知降频
11. ✅ 特殊关怀模式可以生成状态偏离等级
12. ✅ 多指标连续偏离时进入建议照护确认
13. ✅ 特殊关怀模式下运动状态自动降级
14. ✅ 照护闭环状态可以正确更新

## 九、注意事项

1. **当前所有通知均为模拟**，不会真实发送到手机号、邮箱或微信
2. **输入数据仅在本地处理**，不发送到任何服务器
3. **特殊关怀模式是辅助功能**，不替代专业医疗建议
4. **如出现紧急情况**，请直接联系专业医疗机构
