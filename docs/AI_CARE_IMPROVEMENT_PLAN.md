# NeuroGarden AI+疗养系统后续改进策划案

## 1. 项目定位

NeuroGarden 面向“AI+疗养”场景，定位为一个 **个体化情绪状态感知与关怀支持系统**。它不替代医生，不做医学诊断，不预测极端行为，而是通过手机和手表在用户日常使用过程中的低侵入式信号，识别用户相对自身日常节奏的状态偏离，并在合适时机提供温和、可解释、可反馈的陪伴与疗养建议。

项目的核心价值不是“判断一个人有没有病”，而是“在用户可能疲惫、紧绷、低落、孤独或需要支持时，更早、更温和地发现，并提供不打扰、不压迫的帮助”。

## 2. 面向用户

NeuroGarden 面向的是多元人群，因此系统不能只围绕单一画像设计。

### 2.1 心理疾病患者或康复期用户

这类用户可能对压力、低落、惊慌、失眠、孤独等状态更敏感。系统应当避免刺激性表达，不能使用“你有问题”“你很危险”“你抑郁了”等标签化语言。更适合的表达是：

- “我注意到你的节奏有些偏离平时。”
- “要不要先停一下，慢慢呼吸几次？”
- “如果你愿意，我们可以只从一个很小的问题开始。”

对这类用户，准确性和误报控制非常重要。过度提醒可能造成负担，漏报也可能错过支持时机。

### 2.2 朝气满满的年轻人

年轻用户可能并不认为自己需要“疗愈”，但可能存在熬夜、焦虑、压力积累、社交疲惫、考试或工作冲刺等短期状态波动。系统应避免沉重化，更适合提供轻量建议：

- “要不要休息一分钟？”
- “你今天的输入节奏有点急，可以先喝口水。”
- “先把下一次呼气放慢一点。”

对这类用户，系统需要足够轻，不打断工作流，最好通过轻提醒、短呼吸练习、简短对话完成支持。

### 2.3 缺乏陪伴的老人

老年用户可能面临独居、睡眠节律不稳定、身体状态变化、孤独感等问题。他们对复杂交互和权限设置可能不熟悉，因此系统需要：

- 更清楚的权限说明
- 更大的文字和按钮
- 更少的操作步骤
- 可选的守护联系人
- 更多“陪伴感”而不是“检测感”

对这类用户，系统不应过度依赖打字特征，还应更重视手表心率、活动节律、长时间静止、睡眠/清醒变化和主动反馈。

### 2.4 高压工作者和学生

这类用户可能心率、打字速度、删除频率、停顿时长出现明显波动。系统可以重点识别：

- 长时间高强度输入
- 删除频率异常升高
- 呼吸和心率持续偏高
- 深夜高压状态
- 工作后恢复速度变慢

对这类用户，系统要帮助他们从“压力中断”进入“短暂恢复”，而不是要求他们立刻倾诉。

## 3. 当前项目能力

当前版本已经具备以下原型能力：

- 手机端模拟体验
- 个人习惯样本数据库
- 个人基线学习
- 个体化风险评分
- 被动守护前台服务
- 无障碍输入节奏特征采集
- 高风险本地通知
- 守护联系人模拟通知
- 关怀页和呼吸光圈
- 用户反馈写回数据库
- 后端大模型 API 接入
- 轻柔对话与二次风险判断

这些能力已经可以支撑比赛 Demo，但仍然主要是原型系统。尤其在“情绪判断准确性”上，目前还需要重点精进。

## 4. 与理想产品的主要差距

### 4.1 真实生理数据不足

当前手机端心率和呼吸仍存在模拟成分，真实 Wear OS 被动心率、运动状态、活动节律还没有完整接入。没有真实连续数据，情绪判断只能做演示级估计。

### 4.2 情绪标签过粗

当前风险等级主要是 stable、observe、support、guardian_check、urgent_support，更像“风险关注等级”，还不能稳定区分：

- 紧张
- 焦虑
- 低落
- 孤独
- 疲惫
- 运动干扰
- 正常忙碌

后续应从“风险等级”升级为“状态画像 + 置信度 + 解释原因”。

### 4.3 长时间低落判断不足

当前后台是短周期检测，高风险连续出现就提醒。真正的“长时间心情失落”需要趋势判断，例如连续 30 分钟、2 小时、1 天内多次偏离，而不是单点判断。

### 4.4 数据标注不足

系统目前缺少用户主动标注情绪的机制。没有可靠标注，就很难知道模型判断是否准确。后续应加入低负担反馈，例如：

- “刚才的提醒准吗？”
- “现在更像：累 / 烦 / 低落 / 紧张 / 没事”
- “这次提醒太早 / 刚好 / 太晚”

### 4.5 Agent 缺少长期记忆

当前后端大模型可以参与对话和判断，但长期个体化记忆主要还在本地规则侧。后续应让 Agent 使用经过脱敏的摘要和特征趋势，而不是只看一次对话。

## 5. 优先目标：提升情绪判断准确性

目前最优先的目标应该是 **先保证判断准确性，再扩展更多疗养能力**。准确性是这个项目成立的基础。

建议把“准确性”拆成四个层次：

### 5.1 是否真的异常

先判断用户是否偏离自己的日常状态。

输入信号：

- 心率相对个人基线的偏离
- 呼吸频率相对个人基线的偏离
- 打字速度相对个人基线的偏离
- 删除频率异常
- 停顿时长异常
- 运动状态干扰
- 当前时段
- 最近恢复速度

输出：

- 正常
- 轻微偏离
- 明显偏离
- 持续偏离

### 5.2 异常是不是由运动或场景造成

很多误判来自运动、走路、赶路、游戏、会议、考试等场景。系统需要先排除明显干扰。

改进方向：

- Wear OS 运动状态接入
- 手机加速度/活动识别
- 是否正在运动
- 是否处于通勤时段
- 是否正在高频输入或游戏场景

如果是运动导致心率升高，应降低情绪风险判断置信度。

### 5.3 更可能是哪类状态

在确认状态偏离后，再判断偏离类型：

- 高唤醒压力：心率升高、呼吸急、删除频率高、输入急促
- 焦虑紧张：心率波动、停顿和删除交替升高、呼吸偏快
- 低落疲惫：输入速度下降、停顿变长、活动减少、恢复慢
- 孤独陪伴需求：长时间静默、夜间活跃、主动表达孤独或需要陪伴
- 运动干扰：心率高但运动强度也高，情绪置信度降低

输出不应是“诊断”，而是：

- “更像高压状态”
- “更像疲惫低落”
- “更像运动干扰”
- “暂时无法判断，需要更多样本”

### 5.4 当前是否需要介入

不是所有异常都需要提醒。系统应判断是否值得打扰用户。

介入条件可以综合：

- 风险分
- 持续时长
- 当前时间
- 用户历史反馈
- 用户是否经常认为误报
- 最近是否已经提醒过
- 是否开启守护模式

输出：

- 不提醒，只记录
- 轻提醒
- 进入关怀页
- 建议联系守护人

## 6. 情绪判断准确性改进路线

### 阶段 A：真实数据接入

目标：减少模拟成分，让判断建立在真实信号上。

任务：

- 接入 Wear OS Health Services Passive Monitoring
- 获取真实心率
- 获取运动状态或活动类型
- 手表端定期通过 Wear Data Layer 发给手机
- 手机端写入 HabitSample
- 保留模拟模式作为演示和测试入口

交付：

- 真实手表数据链路
- 心率/运动样本入库
- 真实数据和模拟数据可区分

### 阶段 B：低负担用户标注

目标：让系统知道自己判断得准不准。

任务：

- 在提醒后加入“准 / 不准 / 太早 / 太晚”
- 在关怀页加入轻量情绪选择：累、烦、低落、紧张、没事
- 将反馈写入 HabitSample 或新增 FeedbackRecord 表
- 统计误报率和有效提醒率

交付：

- 用户标注数据
- 判断准确率统计
- 个体化阈值优化依据

### 阶段 C：状态分类模型

目标：从单一风险分升级为多状态判断。

建议新增：

```kotlin
data class EmotionalStateEstimate(
    val primaryState: String,
    val confidence: Float,
    val arousalScore: Float,
    val valenceScore: Float,
    val fatigueScore: Float,
    val lonelinessScore: Float,
    val stressScore: Float,
    val interferenceReason: String?,
    val explanation: String
)
```

说明：

- arousalScore 表示唤醒程度，区分平静和紧绷。
- valenceScore 表示情绪倾向，但只能通过弱信号估计。
- fatigueScore 表示疲惫或低活跃可能性。
- lonelinessScore 表示陪伴需求可能性，必须谨慎使用。
- stressScore 表示压力关注程度。

交付：

- 多维状态估计
- 可解释原因
- UI 展示“可能状态”而不是诊断标签

### 阶段 D：趋势窗口

目标：判断“长时间失落”而不是单次异常。

建议窗口：

- 5 分钟：即时状态
- 30 分钟：短期压力
- 2 小时：持续偏离
- 24 小时：日内趋势
- 7 天：习惯变化

任务：

- 新增 TrendAnalyzer
- 计算连续偏离时长
- 计算恢复速度
- 计算日内异常次数
- 识别长期低活跃和长停顿趋势

交付：

- 长时间低落/疲惫判断
- 减少单点误报
- 更合理的提醒时机

### 阶段 E：Agent 复核机制

目标：让大模型参与判断，但不让大模型独自决定。

原则：

- 本地模型先给出结构化特征和初步判断
- Agent 只接收必要特征、趋势摘要、用户主动对话
- Agent 输出建议和解释
- 最终提醒策略由本地规则兜底

Agent 输入示例：

```text
最近 30 分钟：
- 心率高于个人基线 18%
- 呼吸频率高于个人基线 25%
- 删除频率为平时 2.3 倍
- 停顿时长增加
- 运动状态低
- 用户最近两次反馈“提醒有帮助”
```

Agent 输出：

```json
{
  "state": "support",
  "likelyPattern": "高压紧张",
  "confidence": 0.72,
  "suggestedQuestion": "现在更像是累、烦，还是有点绷着？",
  "shouldNotifyGuardian": false,
  "reason": "多项指标持续偏离，且运动干扰低"
}
```

交付：

- Agent 复核准确性提升
- 更自然的轻柔对话
- 降低误报

## 7. 疗养能力改进路线

在情绪判断准确性稳定后，再强化疗养能力。

### 7.1 个体化关怀方案

根据不同状态提供不同支持：

- 高压紧张：呼吸节奏、短暂停顿、肩颈放松
- 焦虑急促：延长呼气、减少问题、提供稳定语句
- 低落疲惫：温和陪伴、小行动建议、光照/喝水/起身
- 孤独陪伴：轻对话、守护联系人选项、日常问候
- 老年用户：更慢节奏、更少文字、更明确按钮

### 7.2 守护联系人真实流程

任务：

- 联系人授权
- 通知前二次确认
- 模拟通知升级为真实短信或服务端通知
- 通知记录
- 随时关闭和撤销授权

### 7.3 关怀记忆

任务：

- 保存用户偏好的关怀方式
- 保存哪些提醒有帮助
- 保存用户不喜欢的表达
- 生成个人化关怀语气

## 8. 隐私与伦理边界

这是项目能否成立的关键。

必须坚持：

- 不使用前置摄像头
- 不保存第三方 App 文本
- 无障碍服务只提取特征
- 用户主动输入给 AI 的内容才可发送给后端
- API key 不写入仓库
- 用户可关闭被动守护
- 用户可删除历史和习惯样本
- 所有守护联系人提醒必须提前授权
- 避免“诊断”“治疗”“预测”等医学化表述

建议新增：

- 隐私说明页
- 数据删除页
- 无障碍服务解释页
- AI 对话数据使用说明
- 本地/云端数据边界说明

## 9. 推荐开发优先级

### P0：情绪判断准确性

- 真实 Wear OS 心率和运动接入
- 用户轻量标注
- 多状态估计模型
- 趋势窗口
- 误报率统计

### P1：被动守护体验

- 更稳定的前台服务
- 通知频率控制
- 低打扰提醒策略
- 权限引导页

### P2：AI 对话与疗养

- Agent 复核
- 更细腻的轻柔问题
- 对话摘要
- 个体化关怀方案

### P3：守护联系人和长期产品化

- 真实联系人授权
- 数据删除
- 隐私合规
- 老年模式和无障碍 UI

## 10. 比赛演示建议

比赛中不要把项目表述为“诊断心理疾病”，而应表述为：

> NeuroGarden 是一个 AI+疗养方向的个体化情绪守护系统。它通过手机和手表在日常使用中的特征信号，学习用户自己的稳定节奏。当系统检测到持续偏离时，会以低打扰方式发起关怀提醒，并通过轻柔 AI 对话进一步确认用户状态，提供呼吸引导、陪伴支持和守护联系人提醒建议。

演示流程：

1. 展示用户开启守护模式。
2. 展示系统学习个人基线。
3. 展示打字节奏、心率、呼吸出现偏离。
4. 展示系统不是直接下结论，而是给出“状态偏离 + 置信度 + 原因”。
5. 展示 AI 发起轻柔对话。
6. 展示用户反馈后系统更新判断。
7. 展示关怀方案和守护联系人模拟提醒。

## 11. 下一步落地建议

下一步最应该先做：

1. 接入真实 Wear OS 被动心率和运动状态。
2. 新增轻量情绪标注。
3. 新增趋势窗口和持续偏离判断。
4. 将风险判断从单一分数升级为多维状态估计。
5. 增加准确率、误报率、有效提醒率统计。

只有先把“判断准不准”做好，后面的 AI 疗养、陪伴、守护联系人提醒才会真正有价值。
