# NeuroGarden 情绪判断准确性 P 级改进策划案

## 1. 目标

本轮改进的核心目标是：让 NeuroGarden 从“根据少量指标判断状态异常”升级为“基于手机端、手表端、个人基线、上下文和 LLM 结构化分析，比较准确地判断用户当前情绪状态”。

这里的“准确”不是医学诊断意义上的准确，也不是判断用户是否患有某种心理疾病。系统只做以下判断：

- 当前状态是否偏离用户个人日常节律
- 这种偏离更接近哪类情绪或体验
- 系统有多大把握
- 是否需要轻柔提醒、继续观察或守护确认
- 用户反馈后，下次判断是否能变得更贴近本人

本轮完成后，系统应至少能区分：

- 平静
- 专注
- 轻松
- 积极活跃
- 疲惫
- 紧张
- 烦躁
- 低落
- 孤独
- 空落
- 压力偏高
- 状态混合或不确定

系统不应只输出“正常 / 异常”，也不应把所有负向波动都归为“难过”。

## 2. 总体原则

### 2.1 个体化优先

同样的心率、打字速度或停顿时长，对不同用户意义不同。所有判断必须优先比较“当前状态”和“该用户自己的历史基线”，而不是使用统一绝对阈值。

### 2.2 多源证据优先

单一数据源不能直接决定情绪。比如心率升高可能来自运动、咖啡、紧张、兴奋或天气。系统需要同时参考：

- 生理数据
- 输入节奏
- 使用场景
- 时间段
- 运动干扰
- 天气
- 用户历史反馈
- Agent 结构化分析

### 2.3 置信度优先

每次情绪判断都必须有 `confidence`。低置信度时，只能展示“可能状态”或“建议观察”，不能强弹窗、不能通知监护人。

### 2.4 隐私边界优先

系统默认不采集、不保存、不上传用户输入原文、聊天内容、视频标题或私密文本。LLM 只接收结构化特征、偏离百分比和用户主动反馈。

### 2.5 非诊断表达

系统不得使用“抑郁诊断”“自残预测”“精神疾病判断”等表达。推荐使用：

- 状态偏离
- 节律波动
- 情绪线索
- 需要关注
- 轻柔支持
- 守护确认

## 3. P0：先保证当前情绪判断链路可靠

P0 是本轮必须完成的底线。完成 P0 后，系统才能说“具备可演示的当前情绪判断能力”。

### 3.1 接入和使用的数据

手机端必须接入并进入情绪判断：

- 最近一次输入速度
- 最近一次删除率
- 最近一次停顿时长
- 最近一次输入样本数量
- 最近前台应用类别
- 当前时间段
- 是否夜间或深夜
- 用户最近一次情绪反馈
- 最近一次 Agent 判断结果

手表端或模拟手表必须接入并进入情绪判断：

- 心率
- 呼吸频率
- 运动干扰 `motionLevel`
- 数据来源：真实手表或模拟手表
- 数据更新时间

环境和系统状态必须接入：

- 天气类型
- 温度
- 湿度
- 数据可信度 `dataQuality`
- 当前 CareMode
- 当前阈值配置 `threshold_profiles`

### 3.2 数据清理

P0 必须增加或明确一层 `SignalPreprocessor`，负责在风险计算和 Agent 调用前统一处理原始信号。

最低要求：

- 过滤明显异常心率，例如小于 35 或大于 220 的值
- 过滤明显异常呼吸频率，例如小于 6 或大于 40 的值
- 对输入速度、删除率、停顿时长做上下限裁剪
- 对运动状态进行修正：高运动干扰时降低心率异常权重
- 对缺失数据打标，而不是补成正常值
- 统一生成最近 5 分钟和最近 15 分钟摘要
- 统一计算相对个人基线的偏离百分比

建议输出结构：

```json
{
  "cleanedSignals": {
    "heartRate": 107,
    "breathRate": 19,
    "typingSpeed": 64,
    "deleteRate": 0.18,
    "pauseDuration": 4.2,
    "motionLevel": 0.22
  },
  "deviations": {
    "heartRatePct": 0.28,
    "breathRatePct": 0.12,
    "typingSpeedPct": -0.18,
    "deleteRatePct": 0.45,
    "pauseDurationPct": 0.31
  },
  "quality": {
    "level": "medium",
    "missingFields": ["hrv"],
    "motionInterference": "low"
  }
}
```

### 3.3 本地情绪初判

在调用 LLM 前，系统需要先做本地情绪初判，避免完全依赖模型。

最低需要支持以下本地规则：

- 高心率 + 高删除率 + 长停顿 + 低运动干扰：倾向紧张、烦躁或压力偏高
- 输入速度下降 + 停顿变长 + 深夜：倾向疲惫、低落或空落
- 输入速度稳定 + 心率平稳 + 删除率低：倾向平静或专注
- 心率略高 + 输入速度高 + 删除率不高：倾向积极活跃或投入
- 数据不足：输出不确定，不强提醒

本地初判输出：

```json
{
  "localPrimaryEmotion": "紧张",
  "localCandidates": ["烦躁", "压力偏高"],
  "localConfidence": 0.62,
  "mainSignals": ["心率偏高", "删除率偏高", "停顿变长"],
  "dataLimits": ["缺少 HRV", "缺少睡眠数据"]
}
```

### 3.4 LLM 调整

P0 必须把 Agent 从“风险判断器”升级为“结构化情绪分析器”。

Agent 输入只能包含结构化数据：

- 清理后的信号
- 相对基线偏离
- 数据可信度
- 时间段
- 天气
- 前台应用类别
- CareMode
- 最近反馈摘要
- 性格/沟通偏好模型

Agent 不允许接收：

- 输入原文
- 聊天内容
- 视频标题
- 私密文本
- 未授权原始健康数据

Agent 输出必须固定为 JSON：

```json
{
  "primaryEmotion": "烦躁",
  "secondaryEmotions": ["紧张", "疲惫"],
  "emotionFamily": "高唤醒负向",
  "valence": -0.55,
  "arousal": 0.72,
  "stress": 0.68,
  "fatigue": 0.46,
  "loneliness": 0.2,
  "riskScore": 0.63,
  "riskLevel": "medium",
  "confidence": 0.71,
  "observedClues": [
    "心率较个人基线升高",
    "删除率升高",
    "输入停顿变长"
  ],
  "counterEvidence": [
    "运动干扰较低，但缺少 HRV 佐证"
  ],
  "uncertainty": "无法确认具体原因，只能判断当前状态相对日常节律出现偏离",
  "suggestedAction": "gentle_popup",
  "careMessage": "我注意到你刚才的节奏有点紧。要不要只用一个词说说现在的感觉？"
}
```

### 3.5 UI 交付标准

今日页必须展示：

- 当前识别到的主要情绪
- 次要情绪
- 置信度
- 情绪维度：愉悦度、唤醒度、压力、疲惫、孤独
- 主要证据
- 不确定性说明
- 数据可信度

弹窗必须展示：

- 温和关怀文案
- 不直接贴负面标签
- 不说“你抑郁了”“你有问题”
- 允许用户反馈“不是这样”

设置或 Debug 页必须展示：

- Agent 是否成功调用
- Agent 延迟
- fallback 状态
- 最近一次 Agent 输入摘要
- 最近一次 Agent 输出摘要

### 3.6 P0 验收标准

完成后应能演示：

1. 开启模拟手表数据
2. 开启后台守护
3. 触发一组高心率、高删除率、长停顿样本
4. 系统显示当前情绪为“紧张 / 烦躁 / 压力偏高”之一
5. 页面展示置信度和判断证据
6. Agent 审计日志中能看到结构化请求和响应
7. 用户反馈“不是烦躁，是疲惫”后，下一次判断候选情绪发生变化

## 4. P1：提高真实场景准确性

P1 的目标是减少误判，让系统在不同场景下更像“理解当前处境”，而不是机械看数值。

### 4.1 新增数据

手机端新增：

- 最近 15 分钟 App 类别变化次数
- 最近 15 分钟亮屏时长
- 最近 15 分钟解锁次数
- 当前是否长时间连续使用
- 当前是否深夜使用
- 通知交互状态：忽略、清除、点击

手表端新增：

- HRV
- 步数或活动强度
- 睡眠摘要
- 静息心率基线
- 运动后恢复速度

### 4.2 数据清理

P1 需要做场景化清理：

- 运动中高心率不直接判为紧张
- 刚解锁后的短暂输入波动不直接判为异常
- 游戏或视频场景下减少输入节奏权重
- 聊天场景下增加删除率、停顿时长权重
- 深夜场景下增加疲惫、空落、低落候选权重
- 连续高强度输入后增加压力和疲惫候选权重

### 4.3 LLM 调整

Agent 增加两阶段推理字段：

```json
{
  "observedClues": [],
  "contextHypothesis": [],
  "counterEvidence": [],
  "dataLimits": [],
  "emotionJudgement": {}
}
```

要求模型先列证据和反证，再输出情绪判断，避免模型直接给结论。

### 4.4 P1 验收标准

完成后应能区分：

- 运动导致的心率升高
- 聊天中压力导致的删除率升高
- 深夜疲惫导致的停顿变长
- 专注工作导致的高输入速度
- 娱乐兴奋导致的轻度心率升高

## 5. P2：建立可学习的用户情绪模型

P2 的目标是让系统越用越懂用户。

### 5.1 新增数据表

建议新增 `emotion_evaluation_records`：

```text
id
createdAt
predictedPrimaryEmotion
predictedSecondaryEmotions
userCorrectedEmotion
confidence
valence
arousal
stress
fatigue
loneliness
signalSummary
contextSummary
agentVersion
wasAccepted
notes
```

### 5.2 用户反馈类型

用户可以反馈：

- 判断准确
- 情绪不对
- 强度太高
- 强度太低
- 我现在是另一种情绪
- 不想被提醒
- 这次提醒有帮助

### 5.3 学习方式

不建议一开始做真正的模型微调。优先做轻量可控学习：

- 统计用户常纠正的标签
- 统计不同场景下的真实反馈
- 更新用户沟通偏好
- 更新情绪标签映射
- 更新本地阈值权重
- 将反馈摘要传给下一次 Agent

### 5.4 P2 验收标准

完成后应能看到：

- 用户多次纠正后，系统候选情绪排序改变
- 用户多次标记误报后，相似场景减少弹窗
- 用户多次确认某类提醒有用后，相似场景提醒更精准
- 设置页或 Debug 页能看到情绪判断命中率趋势

## 6. P3：长期准确性和论文级能力扩展

P3 是比赛后或长期产品化阶段。

### 6.1 更丰富的数据

可考虑接入：

- 睡眠阶段
- HRV 趋势
- 皮肤温度
- 血氧
- 日程压力
- 地理场景粗分类
- 蓝牙设备状态
- 耳机使用状态
- 长期社交活跃度

这些数据都必须经过用户授权，并尽量只保存结构化摘要。

### 6.2 模型评估集

建立本地评估集，而不是盲目微调：

- 典型平静样本
- 典型专注样本
- 典型疲惫样本
- 典型烦躁样本
- 典型紧张样本
- 典型低落样本
- 典型孤独样本
- 运动干扰样本
- 深夜误报样本
- 用户纠正样本

每个样本记录：

- 输入结构化数据
- 用户真实反馈
- 本地算法输出
- Agent 输出
- 最终系统决策

### 6.3 是否微调

只有当满足以下条件时，才考虑微调或训练小模型：

- 已有足够多用户授权样本
- 样本中不含原始私密文本
- 已建立评估集
- 当前 Prompt 和规则方案明显到达瓶颈
- 能做离线评估和回滚

比赛阶段建议优先做：

- Prompt 结构化
- 输出 Normalizer
- 用户反馈闭环
- 本地阈值调参
- 小规模评估集

## 7. 本轮推荐实施顺序

### P0-1：新增 SignalPreprocessor

目标：所有传给风险计算和 Agent 的数据先经过清洗、裁剪、归一化和质量评估。

交付：

- `SignalPreprocessor`
- `CleanedSignalSnapshot`
- 基线偏离百分比
- 缺失字段和数据质量说明

### P0-2：升级 Agent JSON Schema

目标：Agent 输出多元情绪、情绪维度、证据、反证、不确定性和建议动作。

交付：

- 更新 `GuardianAgentApi`
- 更新 `RealGuardianAgentApi`
- 更新 `AgentResponseNormalizer`
- 增加 `EmotionLabelNormalizer`

### P0-3：更新今日页情绪展示

目标：用户能直接看到当前识别情绪，而不是只看到风险分。

交付：

- 主要情绪
- 次要情绪
- 置信度
- 多维情绪指标
- 判断证据
- 不确定性说明

### P0-4：更新弹窗策略

目标：弹窗更轻柔、更人性，不把用户强行归类为负面情绪。

交付：

- 低置信度不强弹
- 中风险轻提醒
- 高风险温和确认
- 支持用户纠正情绪

### P0-5：记录用户纠正

目标：用户反馈能影响下一次判断。

交付：

- 情绪反馈记录
- 最近反馈摘要进入 Agent
- 相似场景候选情绪排序微调

## 8. 判断是否“比较准确”的标准

本轮完成后，可以用以下标准验收：

### 8.1 功能准确

- 当前情绪在今日页可见
- 情绪不只有正负或正常异常
- Agent 输出结构稳定
- 低质量数据不会强行判断
- 用户反馈能记录并影响后续判断

### 8.2 解释准确

- 每次判断都有主要证据
- 每次判断有反证或不确定性
- 用户能理解为什么系统这样判断
- 系统不会说得过满

### 8.3 场景准确

- 运动场景不轻易误判为焦虑
- 深夜长停顿更容易识别为疲惫或空落
- 聊天高删除率更容易识别为紧张或烦躁
- 专注输入不会被误判为异常

### 8.4 反馈准确

- 用户纠正后，下一次相似场景判断会变化
- 多次误报后提醒减少
- 多次确认后相似场景权重提高

## 9. 本轮不建议做的事

本轮不建议：

- 默认采集聊天原文
- 默认读取视频标题或网页内容
- 直接做医学诊断
- 直接微调后端大模型
- 把所有判断交给 LLM
- 在低置信度下强提醒监护人
- 用统一阈值判断所有用户

## 10. 最终交付物

本轮完成后，应交付：

- 更可靠的数据清理层
- 更完整的多元情绪 Agent Schema
- 更稳定的情绪标签归一化
- 今日页当前情绪展示
- 弹窗中的轻柔情绪确认
- 用户情绪纠正记录
- Agent 审计日志增强
- 一组可复现的情绪判断验收样本

最终目标不是让系统“百分百知道用户怎么了”，而是让它能更稳定地说：

> “根据你授权的数据，我看到你现在的节奏和你平时有些不同。它更像是紧张、疲惫或烦躁中的某一种，但我还不确定。你愿意告诉我，现在更接近哪一种吗？”

这才是 NeuroGarden 在 AI+疗养场景中应当追求的判断方式：谨慎、个体化、可解释、可反馈，并且足够温柔。
